# 整体架构

## 产品定位

`vv` 是一款"前门统一、内部分工"的 AI 代理应用：每一次用户请求都进入同一个 **Primary Assistant**，由它自行决定如何回应：

1. **直答** —— 闲聊、定义、计算等不需访问项目的请求。
2. **只读探查** —— 调取代码文件、外部网页等只读资源后给出答案。
3. **委派给专家** —— 把代码改写、研究、审查等任务交给具体的子代理。
4. **多步规划** —— 当任务跨多个能力域时，触发 DAG 编排把工作并行/串行展开。

旧的 `intent → execute → summarize` 三段管道已彻底废弃。`vv` 不再做"前置意图分类"，所有路由决策都由 Primary 通过工具调用完成。

## 分层

```
              ┌─────────── 应用入口 ────────────┐
              │   CLI    │   HTTP   │   MCP    │
              └────┬─────┴────┬─────┴────┬─────┘
                   ▼          ▼          ▼
                       装配中心 (setup)
                          │
           ┌──────────────┼─────────────┐
           ▼              ▼             ▼
        分发器         注册表         记忆/会话
        (Primary +     (代理 +        (memory +
         Dispatcher)   工具能力)        session/tree)
                          │
                          ▼
                   vage 框架 + aimodel SDK
```

- **应用入口**层只负责命令行解析与运行模式选择，不持有业务逻辑。
- **装配中心**是所有可选子系统的接线点，决定哪些组件被构造、按什么顺序拼装。
- **分发器**对外是单一的 `agent.StreamAgent`；对内只做"转发到 Primary 或 fallback"两件事。
- **注册表**把"代理类型有哪些"和"它们能用哪些工具"建模成数据结构，让上层不再硬编码。
- **记忆/会话**是可选子系统，按配置选择性挂载。

## 三种运行模式

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| **CLI** | 终端交互 | Bubble Tea TUI，需要交互的工具（ask_user、权限确认）走对话框 |
| **HTTP** | 远程客户端 / Web 前端 | REST + SSE，无终端；需要交互时走异步回调 |
| **MCP** | 上游 LLM IDE | 把 vv 的代理暴露为 MCP 工具，自身不直接对话 |

`-p` 单提示与 `-eval` 评测是 CLI 系下的两种短期任务模式，只跑一次然后退出。

模式选择遵循 **CLI 标志 > 环境变量 > 配置文件 > 默认值** 的优先级。

## 一次请求的生命周期

1. 用户请求进入应用入口，初始递归深度为 0。
2. 分发器检查深度：超过上限 → 切换到 **fallback Primary**（无工具，避免再触发递归）；否则正常进入 **Primary**。
3. Primary 是一个 ReAct 循环：每轮调用 LLM，然后挑选一个动作执行（直答、读取、委派、规划、协作记笔记/更新树等）。
4. 当 Primary 选择"委派"或"规划"，对应工具会先把递归深度 +1，然后调用子代理或启动 DAG。子代理本身也是 ReAct 循环，结果以工具结果的形式回到 Primary，由 Primary 折叠为最终回复。
5. 整个过程产生的事件（LLM 调用、工具调用、阶段进出）经过统一事件总线分发给：
   - 流式输出（SSE / TUI 渲染）
   - 可选子系统：trace 日志、session 持久化、预算计数、调试 sink。

## 设计原则

### 1. 统一前门

对外只暴露一个 `agent.StreamAgent`（分发器）；策略/路由由 Primary 自己以工具调用方式承担。这样：

- 上层（CLI/HTTP/MCP）不需要为每种代理写不同入口。
- 路由可演化：新增专家代理，只是给 Primary 多挂一个 `delegate_to_*` 工具。
- 失败回退路径单一：递归过深就走 fallback Primary。

### 2. 能力分级

代理能用哪些工具，不在代理代码里硬编码，而是声明一个 **ToolProfile**：

| Profile | 能力 |
|---------|------|
| Full | 读 + 写 + 执行 + 搜索 |
| Review | 读 + 搜索 + 执行（不允许写） |
| ReadOnly | 读 + 搜索 |
| None | 无工具 |

这种分级让"同一个 Factory + 不同 profile"产出多种代理，新增工具时只需归类到能力维度，不必逐个代理修改。

### 3. 递归预算

Primary → 子代理 → 子代理可能再次触发 Primary（通过工具结果中的引导，或专家自己再调 ask_user）。为避免无限循环，所有委派路径都通过 `context` 携带 **递归深度**：

- 分发器在入口处统一检查上限。
- 一旦达到上限，强制切换到 **无工具的 fallback Primary**，物理上消除再次递归的可能。
- 这是一种"硬阀门"，比 try/limit 计数更可靠。

### 4. 零成本默认路径

trace、session、session_tree、budget、debug 等子系统都遵循同一条规则：

- **未启用 → 不构造 → 零开销**。
- 启用时才挂载到统一事件总线上。

这让默认安装跑得既快又简单，按需打开高阶能力。

### 5. 协议无关

所有 LLM 流量经过 `aimodel`，OpenAI 与 Anthropic 在 SDK 层就被规一化为统一接口。换 provider 不动业务代码。

## 子系统全景

| 子系统 | 默认 | 用途 |
|--------|------|------|
| Memory（持久化记忆） | 开 | 跨会话长期事实，分共享/私有 namespace |
| Session（会话） | 开 | 对话历史 + 事件流持久化 |
| Plan Workspace | 跟随 session | Primary 写、子代理只读的 plan.md / notes |
| Session Tree | 关 | 长任务的结构化目标-子任务图 |
| Trace | 关 | JSONL 对话追踪 |
| Budget | 按需 | session/daily 级别 token / cost 硬上限 |
| Debug | 关 | LLM 与工具 I/O 的逐次记录 |
| Web Search | 按需 | 公网检索（Tavily / Brave / SearXNG） |
| MCP Server | 跟随模式 | 把代理暴露给上游 |

子系统之间存在少量依赖（如 Plan Workspace 与 Session Tree 共用会话根；Session Tree 要求 Session 已开），由装配中心统一校验。

## 与 vage 的边界

- vage 提供 TaskAgent、内存抽象、ContextBuilder、工具实体、事件总线、session/workspace/tree 持久化等基础能力。
- vv 的工作是 **挑选、组合、配置**：决定哪些代理存在、哪些工具可用、哪些子系统挂上、外部协议如何映射。
- 反过来：vv 不向 vage 注入业务概念，避免基础库被特定应用形态绑死。
