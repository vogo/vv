# MCP 模式

## 形态与定位

`mode: mcp` 把 vv 转成一个 **MCP 服务端**，让上游的 LLM IDE（Claude Code、Cursor 等）以"调用工具"的方式使用 vv 的代理。

| HTTP 模式 | MCP 模式 |
|----------|---------|
| vv 是"应用"，前端用户面对它 | vv 是"工具源"，上游 LLM 面对它 |
| 暴露完整代理对话能力 | 把每个 dispatchable 代理映射为 MCP tool |
| 终用户负责协调 | 上游 LLM 自行决定何时调用 vv |

简而言之：HTTP 是用户用 vv，MCP 是别的代理用 vv。

## 暴露策略

默认行为：所有 dispatchable 代理（coder / researcher / reviewer 等）都被注册为 MCP 工具。可以通过配置 **白名单** 限制只暴露特定子集，例如只把 reviewer 暴露给团队的 review bot，避免给其它代理隐性的写权限。

分发器（即 Primary）默认 **不暴露**——上游 LLM 一般不需要再多一层路由，他们会直接调用具体专家。配置开关允许显式暴露，主要用于"上游希望委托完整 vv 决策"的少数场景。

## 传输

MCP 协议支持两种传输：

- **stdio** —— 默认；适合本地 IDE 启动 vv 子进程。
- **Streamable HTTP** —— 适合远程暴露。

stdio 不存在身份与网络问题。HTTP 必须考虑：

- **绑定地址**：默认仅绑回环；非回环必须配 bearer token，否则启动报错——这条硬约束防止"裸跑公网"。
- **会话超时**：可选；用于自动清理长时间未活动的连接。

## 凭据扫描

MCP 模式特有的安全措施：所有进出 MCP 通道的工具入参与回值都会经过 **凭据扫描**：

- 出站（vv 调用工具时给出的参数）扫描，防止把内部凭据塞进给上游能看到的字段。
- 入站（工具结果返回上游）扫描，防止 vv 内部读到的凭据泄露给上游 LLM 的 prompt。
- 命中后按配置 `log` / `redact` / `block`。
- 用户可附加自定义正则与白名单。

这层扫描专属于 MCP——CLI 与 HTTP 模式假设运行环境是用户自己的，不存在"对外暴露"问题。

## 与 CLI 权限的关系

MCP 模式 **不挂** CLI 权限拦截链：

- 没有终端，弹不出确认对话框。
- 上游 LLM 期望同步、无打断的工具执行。

替代措施：路径白名单、bash 命令分类器、凭据扫描这三层底层防御依然有效，提供了不依赖交互的安全包络。

## 工具映射的语义

每个被暴露的 vv 代理变成一个 MCP tool。MCP 客户端调用该 tool 时：

1. 上游传入"任务描述"作为 tool input。
2. vv 在 MCP server 端构造一次代理 Run。
3. 代理在自己的 ReAct 循环中运行（仍然是完整能力，包括它自身可以调用其它工具）。
4. 最终回答以 tool output 返回给上游 LLM。

注意：上游看到的是 "我调用了一个工具"，不是 "我对话了一个代理"。所以会话状态不会跨多次 tool 调用累加（除非上游显式管理）。

## 与会话子系统的取舍

MCP 模式下默认仍可启用 session 子系统，每次 tool 调用产生独立 session id（或上游传入）。但因为 MCP 调用一般是 stateless 的，会话视图的价值在 MCP 场景里相对有限。这是配置而非硬约束——保持灵活性。

## 设计要点回顾

- MCP 是 vv 的 **被调用方** 形态，对应一种全新的使用模式（agent-by-agent）。
- 默认安全姿态偏严：仅暴露 dispatchable 代理、非回环必须 auth、凭据扫描默认开。
- 不依赖交互层：CLI 权限拦截不挂，安全靠底层规则。
