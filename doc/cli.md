# CLI 模式

## 形态

CLI 是 vv 的默认形态，提供两种使用方式：

- **交互式 TUI**：常驻进程，类似聊天界面，支持斜杠命令与多轮对话。
- **单提示**：`vv -p "..."`，跑一次然后退出，适合脚本化与 CI 集成。

两种方式共用同一套分发器与代理装配，差别只在于"交互层"。

## TUI 的核心设计

TUI 是一个状态机，三个核心状态：

- **idle** —— 等待用户输入。
- **processing** —— 请求已下发到分发器，正在流式接收事件。
- **confirming** —— 工具调用触发了权限确认对话框，等待用户三选一。

交互界面使用 markdown 增量渲染：流式事件一边到达一边把已确定的部分写到屏幕，未确定部分（流尾的不完整 token）放在缓冲区。这样用户能尽快看到响应，又不会因不完整内容产生跳动。

## 斜杠命令

斜杠命令是 TUI 的"控制平面"，让用户在不离开会话的情况下做：

- 历史管理（清屏、压缩历史）。
- 子系统查看（成本、预算、记忆条目、Session Tree）。
- 模式切换（权限模式、会话恢复）。

斜杠命令故意做得"轻"——它们不属于代理能力的一部分，而是用户对 vv 进程的元操作。这种分离让代理提示词不必处理 UI 控制。

## 权限模式

vv 在 CLI 模式下提供四种工具权限策略，从严到松：

| 模式 | 适用场景 |
|------|---------|
| `default` | 日常使用：危险或写操作弹确认 |
| `accept-edits` | 已经信任本次任务的写操作，不再每次确认 |
| `auto` | 完全信任本次会话；几乎不打断 |
| `plan` | 只允许读和搜索；用于"看代码不动"的场景 |

设计上的几个关键决策：

- **本会话粒度的"永久允许"**：不跨会话。这避免一个临时的 yes 永久放权。
- **bash 命令分类器**：bash 不像 read/write 那样有清晰的危险度，由分类器把单条命令归为安全/危险/黑名单三档。用户可以通过配置扩展规则。
- **路径白名单**：与权限模式解耦。即使用户选了 `auto`，路径白名单仍然有效——这是另一道防线。
- **非交互场景退化**：HTTP / MCP / 单提示模式没有终端，遇危险操作直接拒绝，而不是无条件放过。

## 单提示模式

单提示模式有几条专门的简化：

- **无 TUI**：直接把流式输出写到 stdout，诊断信息走 stderr。
- **非交互式 ask_user**：遇到 ask_user 工具调用直接失败，让模型改换策略。
- **不能进入 HTTP/MCP 模式**：它是一种"短期 CLI 任务"，不适合常驻服务。

这个模式存在的目的是 **可脚本化**：能在 CI / pipeline / cron 中调用 vv 完成具体任务（如 review 一段代码、生成一份 changelog），输出走 stdout 方便管道接下游。

## 会话恢复

`--session` 参数允许用户：

- 列出最近的会话历史。
- 用同一 id 继续上一次的对话。
- 强制开新会话。

当前实现的 MVP 仅恢复 id（让事件、记忆、plan 写到同一目录），**不重放历史消息**。完整的"checkpoint + replay"在路线图中。这种 MVP 取舍的逻辑是：id 复用本身已经能让多次执行共享 plan.md / Session Tree / 记忆，已经覆盖大部分实际诉求。

## 与可观测性的整合

CLI 模式额外承担一些"用户可见的可观测性"：

- 实时显示当前轮的 token 累积与估算成本。
- 预算告警（达到 warn percent 时给出 toast 风格提示）。
- Session Tree 渲染（`vv --tree <id>`，与 LLM 看到的视图一致，便于调试）。

这些都来自统一的事件总线，不需要 CLI 自己额外采集。
