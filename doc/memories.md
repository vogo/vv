# 记忆系统

## 三层视图

vv 直接复用 vage 的记忆模型，把记忆划分为三层视图：

| 层 | 范围 | 寿命 |
|---|------|------|
| Working | 当前 turn 的全部消息 | 一轮请求 |
| Session | 本会话历史 | 一次进程或会话生命周期 |
| Persistent | 跨会话长期记忆 | 永久（文件/SQLite 持久化） |

代理在每轮 ReAct 循环中看到的是这三层经过组合 + 压缩后的视图。

## 持久化后端

Persistent 层有两个后端可选：

- **文件后端**：每条 entry 一个 JSON 文件。简单、可读、易于 grep 调试。适合小规模团队/个人。
- **SQLite 后端**：单文件 WAL 模式。并发写更友好；适合多代理频繁更新或长时间运行的服务。

二者通过同一接口暴露给上层，切换不影响代理代码。选择哪种主要看：是否需要并发写（选 SQLite）、是否需要文件粒度审计（选文件）。

## 命名空间策略

记忆按 namespace 分组，分两类：

- **共享 namespace**：跨会话可见。约定的几个名字：项目级、用户级、约定（约束/惯例）、笔记、默认。这些是"长期事实"，多个会话共享。
- **私有 namespace**：会话级；外部 HTTP 端点不可写。

设计目的：

- 让 HTTP API 只能改共享空间，避免前端误改某个会话的私有内存。
- 共享 namespace 的名字 **是约定而非自由文本**，避免"每个代理自己起名"导致的碎片化。

## Session 层的压缩

Session 层有大小上限。一旦超过，使用滑动窗口压缩器把早期对话摘要为一段紧凑文本。设计取舍：

- 保留最新 N 个 turn 不动（"protected turns"），它们对当前推理最重要。
- 早期内容被压缩为摘要，损失细节但保留主线。
- 压缩本身是另一次 LLM 调用（用同一个 LLM 客户端），输入限制在模型上下文的 80% 以内，避免摘要请求自身溢出。
- 用户也可以通过斜杠命令手动触发压缩。

## 持久化记忆的访问者

只有 Coder 默认使用 Persistent 记忆——它的系统提示会渲染当前所有 persistent 条目，让模型在写代码时看到项目级约定。其他代理（Researcher、Reviewer、Primary）默认不读 persistent 记忆，因为它们的工作不需要这种长期偏置，又能避免每轮无谓的 prompt 膨胀。

## 与会话子系统的关系

记忆系统与会话子系统是 **正交关系**：

- 关闭 Session 子系统 → 会话事件流不持久化，但 Persistent 记忆仍然工作。
- 关闭 Persistent 记忆 → 不影响会话事件流。
- 二者共用同一个 vage 记忆抽象，但各自有独立的存储后端与开关。

这种正交让用户可以按需启用：例如希望事件流落盘但不维护长期记忆（适合一次性诊断会话）。

## 演化路径

记忆系统的几个潜在演化方向：

- **向量化检索**：当前是按 namespace + key 直接查；大量记忆条目下需要相似度检索。
- **多用户隔离**：当前同一进程共享一份持久化记忆；多租户场景需要租户维度。
- **TTL 与容量上限**：当前没有自动过期；需要时可以基于条目时间戳实现。

这些都属于"按需扩展"而非"必须有"——绝大多数当前用例（个人开发助手、单团队 review bot）用基础形态就足够。
