# 初始化装配

装配中心是 vv 中 **最重要的层**：它是配置与运行期组件之间的唯一桥梁。一切"按配置组合不同子系统"的复杂度都集中在此，让上下游各自保持简单。

## 设计目标

1. **单一构造点**：每个组件只在装配中心被构造一次，其他模块都通过依赖注入拿到引用。
2. **零成本默认路径**：未启用的可选子系统不构造、不挂事件、不占内存。
3. **顺序明确**：组件之间存在隐含依赖（事件总线必须在 hook 注册前存在，记忆管理器必须在代理工厂前存在等），装配顺序固化在一处。
4. **可替换的接缝**：上层（CLI/HTTP/MCP）通过 Options 注入少量回调（权限拦截、ask_user 实现、debug sink），不需要重写装配本身。

## 装配阶段

```
配置就绪
  │
  ▼
①  路径与运行环境（工作目录、VV.md 项目级提示）
  │
  ▼
②  LLM 客户端（含中间件链：debug → budget）
  │
  ▼
③  记忆子系统（持久化 store + 三层 manager + 摘要器）
  │
  ▼
④  事件总线 + 可选 hook（trace、session）
  │
  ▼
⑤  Plan Workspace（跟随 session）
  │
  ▼
⑥  Session Tree（要求 session 已开 + 显式启用）
  │
  ▼
⑦  代理装配
       ├── 注册表 + 代理描述符
       ├── 为每个专家代理：按 ToolProfile 构造工具集 → 装饰链 → 工厂
       └── Primary 与 Fallback Primary 单独构造
  │
  ▼
⑧  分发器组装（Primary、Fallback、子代理 map、DAG 配置注入）
```

每一阶段失败都会回滚之前已开的资源（关 store、关事件总线），保证半成品不会泄露到运行期。

## LLM 中间件链

每个对外发出的 LLM 请求都按固定顺序被包装：

```
LLM 客户端
  └─ debug 中间件         （cfg.debug = true 时才存在）
       └─ budget 中间件   （任一 budget tracker 启用时才存在）
            └─ 实际调用
```

routing 配置启用时还会构造一个 **独立的 router LLM 客户端**，套上同一条中间件链，让分发器内部的"轻量分类调用"指向更便宜的小模型。

## 工具装饰链

每个代理拿到的工具注册表是经过多层装饰的：

```
原始 ToolRegistry（按 ToolProfile 构造）
  + ask_user / todo_write 注入
  ↓
权限拦截    （仅 CLI 模式下挂）
  ↓
长度截断    （tool_output_max_tokens > 0 时挂）
  ↓
debug 装饰  （cfg.debug = true 时挂；最外层）
  ↓
代理实际调用
```

顺序的关键约束：

- **权限拦截必须在最内层**：它需要看到原始工具名做策略匹配。
- **debug 必须在最外层**：它要记录代理 *实际收到* 的（已截断的）结果。

## Primary 的特殊装配

Primary 不是普通的 dispatchable 代理，它有两层不一样：

1. **后置注入**：Primary 持有的 `plan_task` 工具需要分发器引用（用来执行 DAG），但分发器又需要 Primary 才能完成构造——所以分发器先以"未注入 Primary"状态创建，再回填。
2. **双实例**：除了带完整工具的 Primary，装配中心还会构造一个 **Fallback Primary**：同样的人格与提示词，但完全没有工具，最大迭代次数=1。这是递归预算超限时的硬保险。

## 可选子系统的挂载策略

| 子系统 | 启用条件 | 与其他系统的耦合 |
|--------|---------|-----------------|
| Session | 默认开 | 提供事件总线的下游消费者 |
| Plan Workspace | 与 Session 同步 | 共用会话目录；Primary 装配多注册若干工具 |
| Session Tree | 显式开启 + Session 必须开 | 共用会话目录；Primary 多注册若干工具；可挂 auto-enable 计数 hook |
| Trace | 显式开启 | 事件总线的另一消费者 |
| Budget | 任一硬上限非零 | LLM 中间件链额外一层 |
| Debug | 显式开启 | LLM 中间件 + 工具装饰各加一层 |

未启用时这些组件 **不构造**，不向上层暴露句柄，对应入口（HTTP 路由、CLI 命令、Primary 工具）也不挂。

## Path 与安全约束

装配过程中确定一组 **canonical 工作边界**：

- 工作目录（默认进程 cwd）。
- 允许访问的目录列表（默认工作目录 + 系统临时目录）。
- 危险命令分类器（基于配置的黑/红/白名单 + 内置规则库）。

这些边界一次性写入文件工具与 bash 工具的构造选项，所有代理共享同一约束，不会因为代理选择不同而改变安全包络。

## 与上下游的契约

- **上游**：通过 Options 注入"是否包权限拦截"、"用哪种 ask_user 实现"、"debug sink 是哪个"。其余信息全部从配置中读取。
- **下游**：返回一个聚合句柄，包含分发器、记忆引用、可选子系统句柄、统一的 Shutdown 函数。任何字段都可能为 nil（对应子系统未启用），上游需做 nil 检查。

这种契约让上层入口非常薄：不需要知道"要构造 8 个组件"，只需要拿到聚合句柄、按字段决定挂哪些路由。
