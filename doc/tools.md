# 工具集合

## 来源

vv 不自实现工具实体，所有工具都来自 vage 的工具子包。vv 自己的工作是：

1. 决定某个代理能拿到哪些工具（按 ToolProfile）。
2. 给工具加上一致的 **装饰链**（权限拦截、长度截断、调试记录）。
3. 提供 vv 特有的"高层工具"：委派、规划、Plan Workspace、Session Tree。

## 工具分类

工具按 **来源与可见性** 分四类：

| 类别 | 工具 | 谁能用 |
|------|------|-------|
| 基础文件工具 | 读 / 写 / patch / glob / grep | 由 ToolProfile 决定 |
| 执行工具 | bash | 仅 Full / Review profile |
| 网络工具 | web_fetch、可选 web_search | ReadOnly 及以上 |
| 协作工具 | ask_user、todo_write | 几乎所有非 None 代理 |
| 路由工具 | `delegate_to_*` / `plan_task` | 仅 Primary |
| 持久化工具 | `plan_update` / `notes_*` / `tree_*` | 仅 Primary（启用对应子系统时） |

"路由工具"与"持久化工具"是 Primary 区别于专家的关键标志：专家只能 *读* 这些子系统的视图，写权一律集中在 Primary 手里。

## 装饰链

每个代理实际拿到的工具注册表，都经过统一装饰：

```
原始工具集（ToolProfile 决定）
  ↓
注入协作工具：ask_user / todo_write
  ↓
权限拦截         （仅 CLI 模式下挂；下文展开）
  ↓
长度截断         （限制单次工具输出 token 数）
  ↓
调试装饰         （记录工具入参与回值；最外层）
  ↓
代理实际使用
```

顺序的两条硬约束：

- **权限在内、调试在外**：权限拦截需要看到原始工具名做策略匹配；调试需要记录代理"实际看到"的（已截断的）结果。
- **截断在权限之外**：截断不应影响权限决策，只应影响代理能看到多长的回值。

## 权限模型

CLI 交互模式下，所有工具调用前都要过一道权限关。规则按 **当前权限模式** + **工具本身的危险度** 共同决定：

| 模式 | 读类工具 | 写/编辑 | 安全 bash | 危险 bash | 黑名单 bash |
|------|---------|--------|-----------|-----------|------------|
| `default` | 自动通过 | 询问 | 自动通过 | 询问 | 永远拒绝 |
| `accept-edits` | 自动通过 | 自动通过 | 自动通过 | 询问 | 永远拒绝 |
| `auto` | 自动通过 | 自动通过 | 自动通过 | 自动通过 | 永远拒绝 |
| `plan` | 自动通过 | 全部拒绝 | 全部拒绝 | 全部拒绝 | 永远拒绝 |

- "询问"会弹出三选一对话框：本次允许 / 永久允许 / 拒绝。
- 一旦选过"永久允许"，该工具在本次进程内不再询问。
- 危险度由 bash 命令分类器决定，分类器接受用户级黑/红/白名单扩展。
- HTTP 与 MCP 模式没有终端，整条权限链不挂；安全约束改由更底层的路径黑名单与命令分类器承担。

## 安全边界

工具系统的安全包络由三个独立层次构成：

1. **路径白名单**：所有文件类工具只能访问允许列表内的路径。允许列表在装配阶段一次性确定，所有代理共享。
2. **命令路径黑名单**：bash 专用，对常见破坏性命令（`rm -rf` 等）做硬拦截，独立于路径白名单。
3. **凭据扫描**（仅 MCP 协议）：扫描通过 MCP 协议进出的工具入参/回值，按配置 log/redact/block 命中条目。

这三层是"深度防御"——任意一层都假设其他层可能失败。

## 高层工具的设计取舍

### 为什么委派是工具而不是专门管道？

把"路由到专家"做成 Primary 的一个工具调用，让模型在 ReAct 循环里自己决定何时使用，相对于"前置硬路由"的好处：

- 模型可以在多轮对话中累积信息后再委派，避免基于第一句话仓促分类。
- 委派失败可以被模型在下一轮看到错误后重新决策，不需要外部回退逻辑。
- 增减专家不改 Primary 主提示词。

代价是每轮多一次 LLM 调用——但 Primary 提示词被刻意做小，加上 prompt cache，单价可控。

### 为什么 plan/tree 写权只给 Primary？

这两类持久化数据是会话级的全局视图。如果允许多个专家同时写：

- 容易出现 race（两个并行专家覆盖彼此）。
- 难以维持一致语义（plan.md 应该是当前任务的完整图景）。

让 Primary 独占写权后，专家通过工具结果把建议反馈回来，由 Primary 决定何时落盘。这是一种 **写者唯一** 的设计模式。

### 为什么 todo 是会话级而 plan 是持久化？

Plan Workspace 是任务策略，应该跨进程重启可见。Todo 是当前 turn 的检查清单，跨 turn 都不一定有意义。两者并存，互不替代：用户能在 plan.md 看"长策略"，在工具事件流看"短进度"。

## 演化策略

新增工具的常规步骤：

1. 在 vage 的 `tool/<name>` 子包里实现工具实体。
2. 在 vv 的能力分级里把新工具归入 read / write / execute / search 之一。
3. 任何 ToolProfile 包含该 Capability 的代理自动拿到新工具。

无需改代理代码。这是 ToolProfile 抽象的核心收益。
