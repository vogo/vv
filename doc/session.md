# 会话子系统：Session / Plan Workspace / Session Tree

vv 把三个 vage 子系统组合到同一个会话目录下：

- **Session** —— 元数据 + 事件流持久化。
- **Plan Workspace** —— 任务级"计划文件 + 笔记目录"。
- **Session Tree** —— 长任务的结构化目标-子任务图。

它们共用一个目录根，`Session` 删除时 **一次目录递归删除** 即可清掉所有三个子系统的状态——这是设计上的关键决策，避免跨子系统的协调一致性问题。

## 启用关系

| 子系统 | 默认 | 启用条件 |
|--------|------|---------|
| Session | 开 | 默认开；显式关 → 三者全关 |
| Plan Workspace | 跟随 Session | 不可单独控制 |
| Session Tree | 关 | 显式开 + Session 必须开 |

依赖关系在装配阶段被强制校验：开 Session Tree 但关 Session 直接启动报错，而不是沉默忽略。这避免了用户配置错误下的隐性问题。

## Session 子系统

负责对话历史的持久化。设计要点：

- **元数据 + 事件流分离**：元数据小而频繁更新（last accessed、title），事件流追加写性质。两类负载用不同文件，避免互相影响。
- **异步 hook 写入**：事件不在主路径上同步落盘，而是通过事件总线异步写。代价是关闭进程时需要主动 flush，收益是在线请求延迟与子系统启用与否无关。
- **id-only 恢复**：当前 MVP 复用 id 让记忆/plan/tree 共目录，但不重放对话历史。完整 checkpoint+replay 在路线图中。

设计上没有引入"会话状态机"——会话只是一组按时间顺序写入的事件。任何"当前状态"都可以由事件回放计算得到。

## Plan Workspace

是 vv 引入的一个 **协作语义层**：

```
Primary           ←→  plan.md（任务策略，跨会话持久化）
                  ←→  notes/<name>.md（任务笔记）
专家代理           只读

todo_write          ←→  本 turn 的检查清单（内存级）
```

设计取舍：

- **写权唯一**：只有 Primary 能写。专家通过反馈让 Primary 决定何时落盘。这是"写者唯一"模式，避免多专家并发覆盖。
- **plan vs todo**：两者并存而非替代——plan 是长策略，todo 是短进度。plan 跨会话；todo 仅当前 turn。
- **容量上限**：plan.md 与每条 note 都有大小限制；超限时 LLM 看到一个明确错误，由模型决定如何分拆。这避免了无限增长导致的提示词爆炸。

## Session Tree

为 **长任务结构化记忆** 而设：当一个任务跨多轮、多分支、多并行子任务时，线性的对话历史不再适合做记忆载体。Tree 提供：

- **目标-子任务的层级结构**：模型可以遍历、聚焦、折叠某子树。
- **"折叠"语义（Promotion）**：当某个父节点下子任务过多或全部完成，把它折叠为一段摘要写入父节点，子节点从主视图消失。这用 Tree 的层级形态实现"渐进抽象"——近处看细节，远处看摘要。

折叠器有三档实现：

- **基于压缩器**（默认）：复用滑动窗口摘要器，零额外 LLM 成本。
- **基于 LLM**：质量高但要付费。
- **noop**：仅翻折叠位，不改 summary。

折叠触发器是"Any-of"组合：子节点数过阈值、子树字节数过阈值、或全部子节点完成。具体阈值由配置决定。

## Auto-enable 门控

Session Tree 启用后默认每轮请求都会渲染 tree 到 prompt 顶部。但短对话用不到树：渲染就是浪费。所以提供一个 **门控阈值**：

- 累积到 N 个 agent 完成事件之前，tree 视图不渲染（仍可手动激活）。
- 到达阈值后开始渲染，让真正"长对话"才付出渲染成本。

阈值是进程级计数，重启清零——用简单计数代替持久化复杂度。

## 写树镜像

启用 Session Tree 且打开"分发器写树"开关时，每次 `plan_task` 都会把 plan 镜像为 tree 节点：

- 第一次 plan 创建 goal 根节点。
- 后续 plan 在根下追加子树。
- 失败仅记录告警，不阻塞 DAG。

这让用户在 tree 视图里看到"vv 自己规划过哪些任务"，把模型的内部决策外显为可观测的结构。

## CLI 与 HTTP 入口

CLI 提供：列出会话、按 id 恢复、强制开新会话、按 id 打印 tree（可选包含已折叠节点）。

HTTP 提供完整 REST 视图：会话列表/详情/事件、Plan Workspace 文件读取、Session Tree 节点 CRUD 与折叠操作。

`DELETE /v1/sessions/{id}` 因为共根设计，单一调用清掉全部三套子系统的状态。

## 设计回顾

会话子系统是 vv 中"一致性收益最大"的设计区域：

- 共根目录 → 删除一致性自动达成。
- 写权唯一 → 多代理并发不冲突。
- 启用关系强校验 → 配置错误启动期暴露。
- 默认渐进开启 → 短对话零负担，长对话才挂载。

更底层的协议见 `vage/.doc/session.md`、`vage/.doc/workspace.md`、`vage/.doc/session-tree.md`。
