# 评测子系统

## 用途

评测子系统让 vv 可以 **对自己的代理质量做回归测试**：给一组测试用例，跑一遍分发器，按多个维度打分，输出报告。两个典型场景：

- **本地回归**：升级模型、改 prompt、新增工具后跑一遍历史用例集，看通过率有没有退化。
- **CI 守门**：流水线里跑核心数据集，失败比例超阈值即阻塞合并。

## 数据模型

测试用例使用 JSONL 格式：每行是一个独立的用例，包含输入提示与预期。这种格式有几个特性：

- **流式追加**：可以一边跑业务一边收集"现实用例"。
- **行级容错**：一行解析失败不影响其它用例，失败行被记入报告。
- **diff 友好**：版本控制下能逐用例 review 修改。

## 评测器

vv 内置四种评测器，可在配置里组合：

| 评测器 | 维度 | 实现成本 |
|--------|------|---------|
| latency | 是否在阈值内完成 | 零额外 LLM 调用 |
| cost | 是否在 token 预算内 | 零额外 LLM 调用 |
| contains | 输出是否包含关键词 | 零额外 LLM 调用 |
| llm_judge | 由 LLM 打分 | 每用例多一次 LLM 调用 |

默认组合是 `latency + cost`——都是免费维度，能立刻反映性能与开销退化。`contains` 适合有客观答案的用例；`llm_judge` 适合需要语义判断的开放问题。

未知评测器名在启动期就报错（不容忍配置错字），避免运行到半路才发现"评测器没生效"。

## 执行流程

```
读取数据集
  ↓
构造评测器（按配置组合）
  ↓
按指定并发跑分发器
  ↓
聚合每用例的多维度结果
  ↓
渲染摘要 + 可选写完整 JSON 报告
```

设计要点：

- **复用分发器**：评测跑的是真实分发器，所以"评测通过"等价于"产品代码通过"。
- **超时单独控制**：每用例独立 wall-clock 超时；超时计为失败。
- **解析失败也写入报告**：让评测者看到"某行 JSON 坏了"这种问题，不再是沉默忽略。

## 两种入口

| 入口 | 触发 | 适用 |
|------|------|------|
| CLI | `vv -eval <dataset.jsonl>` | 本地手动跑、CI |
| HTTP | `POST /v1/eval/run` | 远程触发；需要显式启用 |

CLI 入口与 `-p` 单提示一样属于"短期任务模式"——跑完即退出，退出码反映是否通过。

HTTP 入口默认 **关闭**，避免把评测能力暴露给业务调用者。要启用必须显式打开配置里的 `eval.enabled`，这是有意的"opt-in"安全姿态。

## 报告

输出两份内容：

- **摘要**：直接打到 stdout，给开发者快速读：总数、通过、失败、错误。
- **完整 JSON**（可选）：写到文件，包含每用例的多维度评分、原始输出、token 用量。这份文件是 CI 工件，便于跨次比对。

## 与可观测性的关系

评测期间产生的事件（LLM 调用、工具调用）仍然走主事件总线。这意味着：

- Trace 启用时，每一次评测调用都有结构化 trace，可以事后逐用例复盘。
- Budget 启用时，评测会消耗预算——所以大型评测前要确认 budget 配置。
- Debug 启用时，评测会产生大量调试输出，谨慎组合使用。

## 设计取舍

- **不内嵌"标准答案"**：除了 contains 与 llm_judge，其它评测器都是 **过程指标**（速度、成本），而不是 **结果指标**（对错）。这是因为生成式模型的"对错"高度上下文相关，难以通用化。
- **可扩展**：评测器接口是开放的，未来可以加自定义实现（例如"必须调用了某个工具"、"输出长度在区间内"）。
