# 代理设计

## 角色分工

vv 的代理体系由"一前门 + 三专家"构成：

| 代理 | 角色 | 能力面 | 是否 Dispatchable |
|------|------|-------|------------------|
| **Primary** | 前门统一助手 | 直答 / 只读探查 / 委派 / 规划 | 否（仅由分发器直接调用） |
| **Coder** | 编码专家 | 读 + 写 + 执行 + 搜索 | 是 |
| **Researcher** | 研究员 | 读 + 搜索（含公网） | 是 |
| **Reviewer** | 评审员 | 读 + 搜索 + 执行（不允许写） | 是 |
| Planner | 规划描述 | 无工具 | 否 |

`Dispatchable=true` 的代理才会被 Primary 通过 `delegate_to_*` 调用，并通过 HTTP 暴露为子端点。Planner 是一个特例：它本身不接请求，它的描述字段被 Primary 的提示词拼接器消费，用于"告诉 Primary 有哪些专家可用"。

## Primary 的设计

Primary 是 vv 的核心策略持有者。它的设计有三条硬约束：

### 1. 无写权限

Primary 只能读，不能改。任何 mutation 都必须经过 `delegate_to_coder`。这条约束的目的：

- **可解释性**：用户看到的所有改动都来自一个明确的"专家责任人"，而不是"前门顺手改了一下"。
- **可审计性**：写入路径单一，便于权限拦截、调试与回放。
- **避免角色蔓延**：Primary 一旦能写就会被无限期扩展，最终变成超级代理，丢掉前门的轻量定位。

### 2. 提示词紧凑

Primary 的系统提示刻意保持小型（<800 token）。详细工具用法不写在提示里，依赖工具自身的 schema 描述。原因：

- 每一次请求都会发送 Primary 系统提示，臃肿提示直接放大成本。
- 工具描述紧贴工具定义，更新更不容易遗漏。

### 3. 单次澄清

`ask_user` 是允许的，但提示词显式约束"每轮最多一次"。这避免 Primary 用一连串问题代替判断——研究表明频繁打断显著降低用户满意度。

## 专家代理的能力分工

三个专家的职责边界用 ToolProfile 划分：

- **Coder（Full）** —— 唯一持有写工具的角色。所有真正改代码的事都到这里。
- **Researcher（ReadOnly）** —— 用于 Primary 不便自己读全项目时的二级研究。它能跑搜索引擎和读公网资料，但绝对不动文件系统。
- **Reviewer（Review）** —— 能跑 bash，所以可以执行测试、跑 lint，但不能写。它的任务输出通常是"建议下一步"，由 Primary 决定是否进一步派发给 Coder。

这种分工有意制造的"能力鸿沟"是设计核心：Reviewer 不能修复它发现的问题，只能反馈；Researcher 不能改它读到的代码。这迫使一次 mutation 永远经过 Coder，从而保留单一写入责任人。

## Fallback Primary

除了主 Primary，装配中心还会构造一个 **Fallback Primary**：人格与系统提示完全一致，但 **没有任何工具**，最大迭代次数 = 1。它的作用是：

- 递归深度达到上限时，分发器切到 Fallback。
- 由于没有委派/规划/写工具，它绝对不可能再触发递归。
- 用户看到的"人格"不会断档——还是 Primary 在说话，只是这一轮无法做工具操作。

这是一种用 **类型化代理** 实现的硬保险，比"加 if 检查"更可靠。

## 上下文注入

每个代理在每轮请求会看到的"系统级背景"由几层构成：

```
代理基础系统提示
  + 项目级提示 (VV.md 内容)
  + 持久化记忆 (仅 Coder)
  + Plan Workspace 视图 (启用时)
  + Session Tree 视图 (启用时；可被 auto-enable 门控延后激活)
```

这些上下文以"额外 Source"的形式注入到 vage 的 ContextBuilder 管道，所有代理统一处理，不需要每个代理自己写注入逻辑。

值得注意的是 **专家代理只读 plan/tree**：写工具只挂给 Primary。这是为了避免多写者在同一份 plan.md 上互相覆盖。如果专家发现需要更新计划，它会反馈给 Primary，由 Primary 决定是否调用 `plan_update`。

## 演化策略

新增专家时的步骤：

1. 在注册表里加一个 AgentDescriptor，声明 ToolProfile 与系统提示。
2. 装配中心自动为它构造工具集（按 ToolProfile）、注入到 Primary 的 `delegate_to_*` 工具家族。
3. Primary 在下一次 LLM 调用中就能看到新的委派目标——不需要改 Primary 自己的提示词。

新增能力（例如新加一个 `web_search` 工具）：

1. 把工具归类到一个或多个 Capability。
2. 任何 ToolProfile 包含该 Capability 的代理自动获得新工具。

这种"声明式扩展"让能力面演化的代价被压到最小。
